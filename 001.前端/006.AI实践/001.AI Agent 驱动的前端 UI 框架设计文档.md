# 设计背景与目标

融合 SN 与 Chat Agent 现有能力，在其基础上前端需要具备**动态** **UI** **渲染能力**，即前端 UI **不再由前端写死逻辑**，而是由 **Agent 通过协议数据驱动**

设计的核心思路如下：

1. **协议驱动渲染**：通过统一的数据协议（SNP）描述 UI 渲染需求，前端无需硬编码逻辑
2. **插件化扩展**：前端组件以插件形式注册，协议数据指明使用哪个组件及插槽进行渲染
3. **插槽化布局**：页面划分为 `preprocessor | main | postprocessor | sider` 四个核心插槽，各自承担不同语义角色
4. **可维护性**：所有 UI 渲染、状态管理、命令调用均解耦，保证系统长期可扩展
5. **流式渲染**：支持 AI 响应过程中的流式数据更新（delta），提升用户交互体验
6. **状态可追踪**：提供全链路的状态管理（消息、侧边栏、命令、渲染），保证可观测性和可调试性

# 整体架构设计

核心架构分为五层：

1. **协议层（SNP）**
	- 定义前后端统一的数据协议格式，描述消息、UI 渲染需求、命令调用、附件等
2. **数据层（MessageManager + SiderData）**
    - 负责消息流与侧边栏数据的管理，维护所有 UI 状态
3. **控制层（CommandManager）**
    - 提供全局命令分发机制，供 Agent 或插件调用（如打开侧边栏、切换 layout）
4. **布局层（ChatUILayout / SlotRenderer）**
    - 插槽化布局，负责从数据层获取消息，调用渲染层进行展示
5. **渲染层（RendererManager + 插件组件）**
    - 插件注册与动态渲染，根据 `uiInfo` 中的 `subType` 选择组件进行渲染

# 数据协议

SNP 数据协议中，前端关心的核心部分

```json
{
    "sessionId": "",
    "system": {
        "biz": {},
        "status": {},
        "event": {
                "type": "history | heartBeat | streaming | finish | custom",
                "content": {},
                "command": "", // 能力调用，对应 MCP name
                "args": {} // 调用参数，对应 MCP arguments
        }
    },
    "channel": {
        "branchId": "main", // 分支情况
        "sourceSessionId": "",
        "sourceQuestionId": "", // 原分支回答
        "questionId": "",
        "answerId": "",
        "messageId": "",
        "role": "user" | "assistant" | "system",
        "auth": 0, // 0不可见、1只读、2修改
        "uiInfo": {
            "localMessageId": "",
            "layout": "preprocessor" | "main" | "postprocessor" | "sider",
            "type": "ui" | "data" | "error" | "custom",
            "siderId": "xxxx",
            // 用于前端展示，比如 markdown、Card 组件的 id
            "subType": "", // 使用的组件名(ui，如@bohrium/xxx)、传输数据的业务名（data）、报错类型（error）
            // 操作列表，通过 key 从 content 字段取出数据
            "actionList": [{ "key": "", "action": "append" | "delta" | "replace" | "merge" }],
            "response": {}, // human in the loop发送
            "content": {
                "text": "",
            },
        },
        // 返回给前端的一些其他数据，比如：附件、数据等
        "entities": [
            { "type": "file", "id": "f1", "name": "xxx.pdf", "url": "https://xxx", "tag": ["sessionId", "questionId"] },
            { "type": "paper", "id": "p123" }
        ],
        "meta": {
            "source": "" // 当前消息的来源
        }
    }
}
```

## 协议要点

- **统一协议（SNP）** 适配所有请求与响应：
    - `system`：系统级别的消息 + 业务数据
    - `channel`：消息体 + 插槽布局 + UI 渲染信息
    - `uiInfo`：定义 layout 插槽、渲染组件、渲染方式（delta/replace）

## 数据流动

1. 用户输入问题 → 生成用户消息（`role: user`） → `MessageManager` 存储
2. Agent 响应 → 返回流式数据（`role: assistant`）
3. `MessageManager` 解析 `actionList` → 执行 **delta/replace** 更新消息
4. `ChatUILayout` 根据消息的 `uiInfo.layout`，将消息分发到对应插槽
5. `RendererManager` 根据 `uiInfo.subType` 找到对应插件组件 → 渲染
6. 插件触发交互（如点击引用文献） → 通过 `CommandManager` 下发命令 → 触发 `SiderData` 更新 → `sider` 插槽渲染

# 插槽与布局设计

整个界面采用**插件化插槽布局**，由 `AI Agent` 驱动渲染。前端通过开发插件型组件，并在框架中完成注册。`AI Agent` 输出的数据符合**统一协议（SNP）**，前端根据协议自动选择插槽及组件进行渲染

## 插槽划分

- **preprocessor**：用于展示 AI 的前置信息，例如 **AI Thinking、状态提示、加载中展示**
    - 场景：搜索提示、上下文回溯、推理状态展示
    - 插件示例：`@bohrium/common/thinking`、`@bohrium/common/loading`
- **main**：主内容区，展示核心交互结果，通常包括 **Markdown 文本、数据卡片、结构化输出**
    - 插件示例：`@bohrium/common/markdown`、`@bohrium/paper/card`
- **postprocessor**：用于正文结束后追加展示内容，例如 **引用文献列表、相关推荐**
    - 插件示例：`@bohrium/paper/references`
- **sider**：侧边栏区域，用于展示补充信息，通常由左侧触发，例如：
    - 在 preprocessor 中展示搜索提示时，sider 展示搜索过程
    - 在 postprocessor 中展示文献引用时，sider 展示某篇文献详情
    - 插件示例：`@bohrium/paper/paperDetail`
- **actions bar**
    - 操作列表，用于对当前 Q&A 的操作，例如：**点赞、踩、分享**
    - 操作列表由前端开发后，根据返回的数据中确定是否展示某一项

## 插槽渲染机制

- 插槽由 **ChatUILayout** 统一管理
- 根据消息数据的 `uiInfo.layout`，将消息分发至对应插槽
- 插槽内部调用 **RendererManager** 渲染组件

# 基础设施模块设计

## MessageManager

- **功能**
    - 管理消息存储（`session` → `message list`）
    - 处理 `delta` 与 `replace` 更新
    - 预提供撤回、编辑、重试功能
- **数据结构**

```ts
interface Message {
  sessionId: string;
  messageId: string;
  role: 'user' | 'assistant' | 'system';
  uiInfo: UIInfo;
}
```

## SiderData（基于 zustand）

- **功能**
    - 管理侧边栏状态（`展开`/`收起`）
    - 管理当前绑定的数据实体（如 paperId → 文献详情）
    - 与 MessageManager 联动（点击消息 → 更新侧边栏）

## CommandManager

- **功能**
    - 管理全局命令分发
    - Agent → 命令调用（如 `openSider`、`highlightMessage`）
    - 插件 → 命令调用（如 `引用卡片点击 → 展示sider详情`）
- **示例命令**

```ts
command.exec('openSider',{ entityId: 'p123' });
command.exec('highlightMessage',{ messageId, content: 'xxx' });
```

## RendererManager

- **功能**
    - 插件注册：`subType → React 组件`
    - 插件渲染：根据消息 uiInfo 渲染对应组件
    - 插件隔离：支持多版本、按需加载
- **示例**

```ts
RendererManager.register('@bohrium/common/markdown', MarkdownRenderer);
RendererManager.register('@bohrium/paper/references', ReferenceListRenderer);
```

# 运行时机制

## 流式渲染（delta）

- Agent 返回流式数据 → `MessageManager` 逐步合并内容 → 插槽实时渲染

## 侧边栏联动

- 用户点击 `postprocessor` 中的引用 → `CommandManager.exec('openSider', { entityId })` → `SiderData` 更新 → `sider` 插槽展示详情

##  分支会话 (next version)

- 支持 `branchId` 标记，允许用户在同一会话中产生分支（多条回答对比）

## 1. 用户发起问题

```json
{
    "sessionId": "xxxx-xxxx-xxxx",
    "channel": {
        "role": "user",
        "auth": 1, // 0不可见、1只读、2修改
        "uiInfo": {
            "localMessageId": "",
            "layout": "main",
            "type": "ui",
            "subType": "@bohrium/common/markown",

                "text": "如何优化硅基器件的性能以适应生物医学植入？",

        },
    }
}
```

## 2. AI 响应问题

```json
{
    "sessionId": "xxxx-xxxx-xxxx",
    "channel": {
        "branchId": "main", // 分支情况
        "questionId": "xxxx-xxxx-xxxx-yyyy",
        "answerId": "xxxx-xxxx-xxxx-yyyy_zzzz",
        "messageId": "",
        "role": "assistant",
        "auth": 1, // 0不可见、1只读、2修改
        "uiInfo": {
```
