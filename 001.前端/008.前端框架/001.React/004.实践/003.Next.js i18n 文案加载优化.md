# 背景

bohrium 前端项目在 Next.js 中使用 tolgee 管理国际化文案时，发现一个问题：

SSR 页面返回的 HTML 源代码中出现了所有语言的所有文案

![[Pasted image 20260305161212.png]]

导致：

1. 严重影响 SEO：英文状态，源码里面含大量中文，难以让搜索引擎判定页面语言
2. 严重影响网页性能：html 体积太大，传输和渲染都有很大影响

`bohrium-domains/src/locales/tolgee.ts`

`bohrium-next-app/src/locales/tolgee/shared.ts`

即便使用了 `import()` 动态引入，tolgee SSR 仍然加载了所有的资源

# 解决方案

## 第一步 只加载当前语言的文案

拿取 Cookie 中的语言设置，只给 tolgee 传入当前语言的资源

结果：体积少一半（中+英 ➡️ 中/英）；中英不掺杂

副作用：无。由于我们现在网站是切换语言刷新浏览器，所以不会导致文案缺失

![[Pasted image 20260305161302.png]]

## 第二步 只加载指定命名空间的文案

第一步少了一半文案，但仍然包含所有的中或英文案，可以进一步优化为加载使用到的几个命名空间下的文案即可

结果：体积少2/3（30+命名空间文案 ➡️ 3～8个命名空间文案）

副作用：如果没有在服务端加载首页用到的命名空间，会导致刷新页面出现1秒左右无文案状态（显示key）

# 使用指南

bohrium-next-app/src/app 下：

某个路径下公用的命名空间（layout.tsx 中)

```JavaScript
export default async function SharedTabLayout({ children }: { children: React.ReactNode }) {
    await loadNamespaces(['journal']);

    ...
}
```

某个页面的命名空间（page.tsx 中)

```JavaScript
export default async function Page({ params }: PageProps) {
    await initServer();
    await loadNamespaces(['paper-detail']);
    
    ...
}
```

- 只有首页用到的需要加，下滑页面才能看到的不用加（滑过去的时候已经动态加载完了）
- 可以在bohrium-next-app/src/app下的某个路径下加loading.tsx，这样做客户端动态加载完之前会展示loading
- 在 useEffect、useCallback、useMemo 使用到 t('xx') 的地方要把 t 写入依赖数组中，否则页面文案缺失

> Claude Sonnet 4.5：
> 
> 非常好的想法！你提出了一个更深层次的优化需求。让我分析一下自动扫描并只携带使用到的翻译 key的可行性：
> 
> ### 方案 1：构建时静态分析（最优，但复杂）
> 
> 原理：在构建时扫描所有组件，提取使用的翻译 key
> 
> 实现步骤：
> 
> 1. 编写 Babel/SWC 插件扫描 t() 和 getTranslate() 调用
>     
> 2. 生成每个页面的 key 使用清单
>     
> 3. 在 SSR 时只传递使用的 key
>     
> 
> 优点：
> 
> - ✅ 完全自动化
>     
> - ✅ 最精确的优化
>     
> - ✅ 零运行时开销
>     
> 
> 缺点：
> 
> - ❌ 实现复杂度高（需要编写编译器插件）
>     
> - ❌ 动态 key 难以处理（如 t(variableKey)）
>     
> - ❌ 维护成本高
>     
> 
> ### 方案 2：两次渲染 + 运行时追踪
> 
> 原理：先渲染一次收集使用的 key，然后只传递这些 key
> 
> 优点：
> 
> - ✅ 能追踪动态 key
>     
> - ✅ 不需要修改构建流程
>     
> 
> 缺点：
> 
> - ❌ 需要渲染两次（性能开销）
>     
> - ❌ 可能需要修改 Tolgee 源码
>     
> - ❌ Next.js RSC 架构下可能不可行
>     
> 
> ### 方案 3：混合方案（推荐平衡）
> 
> 原理：
> 
> 1. 开发时：手动标记关键页面的 key 列表
>     
> 2. 构建时：自动扫描收集（可选）
>     
> 3. 运行时：动态加载缺失的 key
>     
> 
> ## 我的建议
> 
> 基于收益/成本分析：
> 
> ### 短期方案（推荐）✅
> 
> 继续使用 Namespace 方案，因为：
> 
> 4. 已经减少了 70% 的数据量
>     
> 5. 实现简单，维护成本低
>     
> 6. 对大多数应用来说已经足够优化
>     
> 
> ### 中期方案（可选）
> 
> 针对超大页面做定制优化：
> 
> - 识别翻译数据特别大的 namespace（如 database, journal）
>     
> - 手动拆分成更细的 sub-namespace
>     
> - 或者创建页面专用的小 namespace
>     
> 
> ### 长期方案（如果确实需要）
> 
> 考虑实现静态分析（方案 1），但需要：
> 
> 1. 评估团队技术栈（是否有编写编译器插件的能力）
>     
> 2. 考虑维护成本
>     
> 3. 先做 POC 验证可行性
>     
> 
> ## 我的最终建议
> 
> 4. 先用 Namespace 方案（已实现），它能解决 80% 的问题
>     
> 5. 监控实际效果：测量首屏加载时间、HTML 大小
>     
> 6. 如果仍有性能瓶颈：
>     
> 
> - 考虑拆分超大 namespace
>     
> - 或者为关键页面（首页、落地页）做定制优化
>     
> 
> 4. 只有在有明确收益时，才考虑实现完整的 key 级自动追踪
>

> 
> 你说得对！这确实是一个很常见的需求。让我深入调查一下 Tolgee 和业界的解决方案。
> 
> 好的，我做了全面的调查。以下是结论：
> 
> ## 📊 调查结果
> 
> ### ✅ 确认：Tolgee 没有自动提取使用 key 的功能
> 
> 经过多次搜索和文档查阅，Tolgee 确实不支持自动扫描 HTML/组件并只加载实际使用的翻译 key。它的最小粒度是 namespace。
> 
> ### 🔍 业界现状
> 
> 虽然这是个合理的需求，但业界也没有主流 i18n 库在运行时实现这个功能。原因是：
> 
> #### 1. 技术难点
> 
> - 动态 key：t(variable) 这种运行时决定的 key 无法静态分析
>     
> - 条件渲染：{isShow && <Text>{t('key')}</Text>} 这种条件分支
>     
> - 异步组件：延迟加载的组件使用的翻译
>     
> - SSR/RSC：React Server Components 渲染时机复杂
>     
> 
> #### 2. 现有工具都是"构建时"方案
> 
> 业界有一些构建时的工具来优化翻译：
> 
> ![](https://dptechnology.feishu.cn/space/api/box/stream/download/asynccode/?code=OWM2MGNlMGEwM2ZjZDNlMzQ1Yjk4NTZiZjZiZTE2NTRfS3lleElWWUV4anNHWE50ZUx6UGxxc3U0TlJQMzJRbmNfVG9rZW46U2N2N2JPVXQzb0w4NEp4ZUNQdGNud0Vhbk1TXzE3NzI2OTgzMDk6MTc3MjcwMTkwOV9WNA)
> 
> 这些工具的目的是：
> 
> - ✅ 帮助开发者发现代码中使用了哪些翻译 key
>     
> - ✅ 生成翻译文件（初始化语言包）
>     
> - ✅ 检查是否有缺失的翻译
>     
> 
> 但它们不能：
> 
> - ❌ 在 SSR 时动态判断页面使用了哪些 key
>     
> - ❌ 只在 HTML 中输出使用到的翻译
>     
> 
> ### 💡 为什么没有运行时方案？
> 
> 我理解你的想法非常合理，但运行时自动提取使用 key 有根本性困难：
> 
> _// 场景 1: 动态 key_
> 
> const key = Math.random() > 0.5 ? 'key1' : 'key2';
> 
> t(key); _// 怎么在渲染前知道会用哪个？_
> 
>   
> 
> _// 场景 2: 条件渲染_
> 
> {user.isPremium && <Text>{t('premium_feature')}</Text>}
> 
> _// 不同用户看到不同内容_
> 
>   
> 
> _// 场景 3: 循环生成_
> 
> items.map(_item_ => <div>{t(`item_${item.id}`)}</div>)
> 
> _// key 是动态拼接的_
> 
> ## 结论
> 
> 1. Tolgee 确实没有这个功能（已确认）
>     
> 2. 业界也没有运行时的自动方案（技术限制）
>     
> 3. Namespace 方案已经是业界标准（所有主流库都这样）
>     
> 4. 可以通过拆分 namespace 进一步优化（页面级粒度）
>     
>
